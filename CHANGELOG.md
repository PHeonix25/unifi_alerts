# Changelog

All notable changes to this project are documented here. The format is based on
[Keep a Changelog](https://keepachangelog.com/en/1.1.0/), and the project follows
[Semantic Versioning](https://semver.org/spec/v2.0.0.html).

`docs/HISTORY.md` is the source-of-truth, dated, narrative log; this file is the
user-facing summary scoped to releases. Pre-release entries (`X.Y.Z-preN`) are not
listed individually — only the consolidated `X.Y.Z` release that bundles them.

## [Unreleased]

### Security

- Webhook ID collision on multi-entry config setups (CRITICAL): the new `CONF_WEBHOOK_ID_SUFFIX` (per-entry, generated by the config flow) prevents two config entries from registering on the same webhook ID. Existing single-entry users keep their already-pasted Alarm Manager URLs; multi-entry collisions are now impossible. (`const.py`, `webhook_handler.py`, `config_flow.py`)
- Timing-safe webhook token comparison via `hmac.compare_digest`. (`webhook_handler.py`)
- DEBUG log redaction: `?token=<secret>` scrubbed to `?token=***` in setup logs; webhook payload DEBUG log narrowed to known-safe fields only. (`__init__.py`, `webhook_handler.py`)
- Webhook secret rotation surfaced in the options-flow credentials step ("Regenerate webhook secret" checkbox). (`config_flow.py`)
- Per-(category, alert_key) 5s rate-limit / debounce on inbound webhooks to prevent flood from a misconfigured Alarm Manager. (`coordinator.py`)

### Reliability

- `WebhookManager.register_all()` now wraps each iteration in try/except so a single failed registration cannot abort the rest of the loop. (`webhook_handler.py`)
- Webhook decode failures now log at WARNING with the exception class name and a 80-byte body preview, instead of falling through silently to `{}`. (`webhook_handler.py`)

### Repo hygiene

- `CHANGELOG.md` (this file) added; back-filled from v1.0.0.
- `SECURITY.md` added with vulnerability disclosure guidance.
- `CODEOWNERS` added.
- GitHub bug-report and feature-request issue templates added alongside the existing `unclassified_event_key.yml`.
- Dependabot config (`.github/dependabot.yml`) added for `github-actions` so pinned SHAs surface as PRs on upstream releases.
- `.github/release.yml` (release-notes categories file) added so auto-generated release notes group commits into Bug Fixes / Features / Security / Chores.
- `release.yml` workflow migrated from `softprops/action-gh-release` to GitHub CLI's `gh release create --generate-notes`. Eliminates the only third-party action in the release pipeline; release notes are now scoped to the previous-tag → current-tag window.

## [1.3.0] — 2026-04-29

### Fixed

- Options flow no longer loops between pages 1 and 2; the credentials → categories → finish flow now mirrors initial setup correctly. (`config_flow.py`, `strings.json`, `translations/en.json`)
- Service / device card now appears in HA's Devices & Services page immediately after setup; `configuration_url` added to `_device_info()` helpers. (`__init__.py`, all platform files)
- Message sensor defaults to `"No alerts yet"` instead of blank; `EntityCategory.DIAGNOSTIC` applied to message sensors and `EntityCategory.CONFIG` to clear buttons. (`sensor.py`, `button.py`)
- Removed `EventDeviceClass.BUTTON` from event entities (semantically incorrect for alert events). (`event.py`)
- API-key authentication on UniFi OS controllers no longer misdetects `_is_unifi_os = False`. (`unifi_client.py`)
- HTTP status codes are now surfaced in `CannotConnectError` messages for easier troubleshooting. (`unifi_client.py`)

### Changed

- Alarm endpoint probe chain extended to `[/list/alarm, /alarm, /stat/alarm]` (newest → oldest) to support UniFi Network 9.x+. (`unifi_client.py`)
- Removed the invalid `limit=200` query parameter from `/alarm` calls; controllers with > 200 open alarms no longer silently drop results. (`unifi_client.py`)

### CI

- Pre-release `grep` regex fixed: `--` terminator added to `grep -qE -- '-pre[0-9]+$'` so `vX.Y.Z-preN` tags are no longer parsed as CLI flags and incorrectly published as stable releases. (`.github/workflows/release.yml`)
- `softprops/action-gh-release` bumped from v2 to v3 (Node 24, ahead of the 2026-06-02 Node 20 EOL).

## [1.2.0]

Internal critical-review pass — no user-visible changes published as 1.2.0; the items surfaced in this audit were carried forward into 1.3.0 / 1.4.0. See `docs/ROADMAP.md § v1.2.0` for the audit findings list.

## [1.1.0]

### Added

- Lovelace dashboard YAML example in README.
- Automation example in README with verified `event_type` / `event_data` schema.
- Service calls: `unifi_alerts.clear_category` and `unifi_alerts.clear_all`. (`services.py`, `services.yaml`)
- Config-entry repair flow: a HA repair notification surfaces when auth fails post-setup. (`config_flow.py`)
- Options flow: credentials and controller URL can be updated without re-adding the integration.

### Security

- Controller URL validated for `http(s)` scheme to prevent SSRF via non-HTTP schemes. (`config_flow.py`)
- Webhook body capped at `WEBHOOK_MAX_BODY_BYTES` (8 KB) so malformed POSTs cannot exhaust memory. (`webhook_handler.py`)
- Authentication exception messages no longer log `str(err)` (which can include credential fragments); only the exception class name is logged. (`unifi_client.py`)

### Fixed

- Polling re-auth log message no longer claims success when re-auth succeeds but the second poll fails. (`coordinator.py`)
- Removed `limit=200` cap on `/alarm` endpoint (no pagination loop available; default of 3000 covers all known cases). (`unifi_client.py`)

### CI

- All GitHub Actions `uses:` references pinned to commit SHAs.
- Two-branch model: `main` (stable, `X.Y.Z`) and `dev` (pre-release, `X.Y.Z-preN`). `version-check.yml` enforces the format per branch.

## [1.0.0]

### Added

- First stable release of the UniFi Alerts custom integration.
- Per-category binary sensors, message sensors, open-count sensors, event entities, and clear buttons.
- Rollup binary sensor, count sensor, and rollup message sensor across all enabled categories.
- Config flow with three-step setup: credentials → categories → webhook URL display. Username/password and API-key auth supported.
- REST polling backstop on a configurable interval; webhook push path for real-time updates.
- Per-category bearer-token authentication on inbound webhooks.

### Security

- SSL verification enabled by default (`DEFAULT_VERIFY_SSL = True`). UI toggle for self-signed-cert controllers.
- Webhook secret generated as `secrets.token_urlsafe(32)` at first setup; required as `?token=` query parameter on every inbound webhook.
- Webhooks marked `local_only=True` so HA rejects requests from outside the local network.

### Fixed (pre-1.0 install-testing blockers)

- UCG-Ultra OS detection: two-stage fallback probe (x-csrf-token header + `/api/system`) added.
- Config flow API-key field guidance reworded to be firmware-version agnostic.
- Sensitive credential fields use `TextSelectorType.PASSWORD` so they are not displayed in plaintext.
